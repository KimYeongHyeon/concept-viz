AI 매핑 검토 시각화를 8단계로 체험합니다

각 단계마다 해결하려는 문제, 데모에서 시도해볼 동작, 그리고 인터랙티브 데모가 함께 제공됩니다. 데이터의 구조부터 시작해 점점 복잡한 상호작용을 쌓아가며, 마지막 8단계에서 모든 기법이 하나의 화면으로 통합됩니다.

1

이 투어가 무엇을 보여주는가

Orientation · 1분 읽기
⚠ 해결하려는 문제

임상시험 eligibility criteria를 OMOP concept로 매핑할 때, AI는 후보를 제안하지만 의료진이 검증하는 단계에서 병목이 발생합니다. Athena·Usagi·ATLAS는 단일 concept 조회와 cohort 생성에 특화되어 있어, "AI 후보 검토"라는 중간 워크플로우가 빠져있습니다.

✓ 이 투어에서 보여드릴 것

검색 중심 워크플로우를 시각적 탐색으로 대체하는 세 가지 기둥 — DOI Tree 탐색, 실시간 feasibility 미리보기, anchor 기반 gap 감지 — 그리고 이들을 하나의 화면에 결합한 통합 데모입니다.

데이터의 생김새 — 왜 Tree가 현실적인가

Data foundation · child-histogram
문제

OMOP Vocabulary는 44만 concept입니다. 트리로 펼치면 화면이 터질 것 같지만, 실제로 각 concept가 몇 개의 직계 자녀를 가지는지 분포를 알지 못하면 UX 설계가 불가능합니다.

발견

분석 결과 95%의 concept가 직계 자녀 ≤19개. 즉 한 번에 펼칠 때 대부분 20행 이하로 표시 가능하다는 정량적 근거가 생깁니다.

▶ 시도해보세요
  1. 히스토그램에서 95번째 퍼센타일 위치 확인 → UX 설계 근거
  2. 롱테일에 속한 concept들 살펴보기 (이상치 처리 전략이 필요)

전체 분포 Overview — 탐색의 출발점

Space-filling · Treemap / Sunburst
문제

44만 concept를 처음 보는 의료진은 어디부터 시작해야 할지 막막합니다. SNOMED, RxNorm, LOINC 중 어느 도메인을 우선 볼지 감이 없습니다.

접근

Treemap / Sunburst로 크기로 규모를 보여줍니다. 영역의 면적이 concept 수에 비례 → 어디에 무게중심이 있는지 한 눈에 파악.

▶ 시도해보세요
  1. 가장 큰 영역(vocabulary)을 클릭해 하위 domain 진입
  2. Treemap ↔ Sunburst 전환해 같은 데이터를 다른 각도로 보기
  3. Drill-down 후 다시 root로 되돌아오는 과정의 방향감 유지 확인

Focus + Context 탐색 — 핵심 탐색 기법

DOI Tree · Semantic Zoom
문제

Tree를 일반적으로 펼치면 관심 없는 가지까지 함께 열려 화면이 산만해집니다. 검색으로 특정 노드만 보면 주변 맥락이 사라집니다.

접근

DOI(Degree of Interest)가 높은 가지는 펼치고, 나머지는 축약 — Semantic Zoom으로 줌 레벨에 따라 label이 자동 전환됩니다.

▶ 시도해보세요
  1. 특정 concept 클릭 → 그 주변만 세부 전개, 다른 영역은 축소
  2. 줌 아웃 → concept label이 요약 label로 바뀌는 level-of-detail 전환 관찰
  3. 여러 번 초점 이동 → 이동 경로의 애니메이션으로 맥락 유지 확인

Concept Set 시각적 구성 — Drag-to-Select

Interactive selection · Lasso
문제

기존 도구는 concept을 하나씩 체크박스로 선택합니다. 10~20개 concept을 포함하는 criteria에서는 반복 클릭 피로가 누적됩니다.

접근

드래그·래소로 시각적으로 영역을 그리면 그 안의 concept가 전부 선택됩니다. 선택한 concept set은 오른쪽 패널에 누적되며, 전체 구성이 한 눈에 보입니다.

▶ 시도해보세요
  1. 여러 concept을 드래그·래소로 한 번에 선택
  2. 선택 해제: Shift+드래그 또는 선택 패널에서 X 클릭
  3. 선택된 concept들의 계층 관계가 오른쪽 패널에 표시되는지 확인
demo-interactive-select.html

AI가 놓친 concept 찾기 — Anchor-based Review

Mapping review · Gap detection
문제

AI가 매핑한 concept 목록을 받아도, "놓친 것"은 보이지 않습니다. 의료진이 "여기에 sibling X도 있어야 하지 않나?" 싶어도 확인하려면 Athena를 별도로 열어 수동 탐색해야 합니다.

접근

AI가 선택한 concept을 anchor로 삼아 그 주변(sibling, child)을 자동 전개. 포함된 것은 채워진 셀, 비어있는 것은 점선 셀 — 시각적 gap이 즉시 드러납니다.

▶ 시도해보세요
  1. AI 추천 concept 주변의 점선 셀(포함 안된 sibling) 찾기
  2. 점선 셀 클릭 → 추가 검토 패널 → 임상적 타당성 판단 후 포함 여부 결정
  3. 여러 anchor가 만드는 중첩 영역이 어떻게 표시되는지 확인
demo-mapping-review.html

Feasibility 실시간 미리보기

Threshold slider · Attrition waterfall
문제

기존 워크플로우에서는 매핑이 모두 끝난 뒤 cohort 생성 단계에서야 환자 수가 나옵니다. "이 기준이 너무 빡빡했구나"는 뒤늦게 발견됩니다 → 매핑 재작업.

접근

Threshold 슬라이더(HbA1c > 7.0 → 7.5 → 6.8)를 조정할 때마다 attrition waterfall이 즉시 갱신 — 각 criterion이 얼마나 많은 환자를 탈락시키는지 실시간 확인.

▶ 시도해보세요
  1. HbA1c threshold를 이동 → waterfall의 해당 bar가 실시간 갱신
  2. Age 범위를 넓혀보기 → 최종 cohort 크기 변화 관찰
  3. 어느 criterion이 가장 많은 환자를 탈락시키는지 확인 → 완화 가능성 검토

통합 워크플로우 — Tree ↔ Feasibility 실시간 연동

CORE DEMO · Integrated Review + Feasibility
지금까지의 문제

앞의 7단계에서 본 기법들은 개별적으로는 훌륭하지만, 실제 검토 작업은 이들을 동시에 오가야 가능합니다. Tree 탐색하면서 cohort 크기를 봐야 하고, gap을 발견하면 concept set을 수정해야 하고, 바꾸면 다시 feasibility를 확인해야 합니다.

이 데모가 보여주는 것

왼쪽 Tree에서 concept을 클릭 하나로 교체하면 오른쪽 Funnel이 즉시 갱신 — 탐색 · 선택 · 검증 · 재조정의 단일 루프가 완성됩니다.

▶ 시도해보세요 (이 투어의 마지막 스텝)
  1. Tree에서 sibling concept을 클릭해 현재 선택을 대체 → Funnel의 n이 어떻게 변하는지 확인
  2. 여러 개를 연속으로 교체하며 "조금 넓히니 n이 2배" 같은 trade-off를 즉시 감각
  3. AI가 제안한 원래 매핑과 의료진이 수정한 매핑의 차이가 cohort에 미치는 영향 비교
  4. 이 사이클을 반복해 임상적으로 타당하면서 feasibility도 확보된 매핑에 수렴
